# flutter_rust_net Rust 通道业务适用性分析（2026-03-13）

> 目的：把仓库内已经落地的 benchmark 结果，翻译成“哪些业务接口适合优先切 Rust，哪些接口暂不建议切 Rust”的可执行建议。

## TL;DR（几分钟版）

1. 不要把 Rust 通道理解成“所有请求都应该替换 Dio”；当前证据只支持优先用于“传输效率型”读请求。
2. 最值得优先切 Rust 的是三类：
   - 大包体下载、资源拉取、离线包、媒体分片这类大流量 GET。
   - 大 JSON 读取，但仍走 `bytes/bodyFilePath -> Dart decode/model` 的 bytes-first 路径。
   - 批量只读预拉取、并发读请求，但前提是并发闸门 `maxInFlightTasks >= 32`。
3. 先不要默认切 Rust 的是三类：
   - 公网下首屏关键小请求，因为当前公网短请求样本里 Rust 的首包稳定性还不够稳。
   - 高抖动高并发、但并发闸门还没放开的场景，因为容易放大尾延迟。
   - 非幂等写请求，因为这类接口不适合依赖自动 fallback 兜底。
4. 当前更合理的路由思路是：
   - Rust 负责“大响应、并发读、可灰度、可 fallback”的幂等读接口。
   - Dio 继续负责“首屏短平快、小请求、写操作、高风险接口”。
5. 结论边界要看清：
   - 现有 benchmark 主要证明的是 L1 传输层收益，不等于“Rust 直接解析业务对象回 Dart”也一定更优。
   - 2026-03-13 的公网样本主要还是 `Windows + ethernet + smoke`，还不能直接当作 Android / iOS 真机长期结论。
6. 如果要落地，建议顺序是：
   - 第一批先切：大包体 GET、大 JSON GET、批量只读预拉取。
   - 第二批再看：中等体积读接口，但要补真机和目标网络样本。
   - 最后再评估：公网首屏短请求、小接口、非幂等写接口。

一句话判断：接口越像“幂等读 + 大响应/并发预取 + 可 fallback”，越适合 Rust；越像“首屏小请求 + 写操作 + 对单次抖动极敏感”，越应该先留在 Dio。

## 补充说明：这更像当前库阶段性结论，不是 Rust 语言定论

1. 目前不能把“公网短请求里 Rust 没稳定赢”直接理解成“Rust 天生不适合小请求”。
2. 更合理的解释是：当前结果同时受 Rust 通道实现成熟度、路由策略粗细、初始化与并发参数配置、以及样本覆盖范围影响。
3. 仓库内反而已经有不少反证说明“不是 Rust 本身不行”：
   - `large_payload` 明显是 Rust 领先。
   - `large_json` 在 bytes-first 路径下也是 Rust 领先。
   - `small_json` 在 loopback/L2 样本里 Rust 也能领先。
   - `jitter_latency` 在 `maxInFlightTasks >= 32` 后，Rust 能从排队劣势中恢复。
4. 当前真正还不够强的，更像是“这套库对业务场景的适配还不够细”：
   - 路由策略现在主要还是总开关级别，不是按接口类型、首屏敏感度、响应体大小、并发特征做精细分流。
   - 公网证据目前主要还是 `host_windows + ethernet + public_remote + smoke`，还不能代表 Android / iOS 真机长期表现。
5. 所以本文的正确读法应该是：
   - 现在只能说“在当前库实现和当前证据下，Rust 更适合传输效率型读请求”。
   - 不能说“Rust 从原理上就不适合短请求”。
6. 后续如果补齐真机、公网、弱网、首跑/rerun 对照和更细路由后，结论仍然有可能继续变化。

## 1) 结论先行

1. 当前最适合优先切 Rust 的，不是所有网络请求，而是这几类：
   - 大包体下载、资源拉取、离线包/媒体/大文件类 GET。
   - 大 JSON 读取，但仍采用 `bytes/bodyFilePath -> Dart decode/model` 的 bytes-first 路径。
   - 并发预拉取、批量只读请求、弱网下容易堆积重试的读请求。
   - 可接受灰度、可观测、可自动 fallback 的幂等读接口。
2. 当前不适合直接默认切 Rust 的，是这几类：
   - 公网下极短小、极度看重首包时延的首屏关键小请求。
   - 高抖动并发场景下，如果 `maxInFlightTasks < 32`，Rust 容易因为并发闸门放大尾延迟。
   - 非幂等写请求，或虽然可走 Rust，但不希望依赖自动 fallback 兜底的接口。
3. 当前仓库证据更支持一句话：
   - Rust 适合“bytes-first 传输效率型”接口。
   - Dio 仍更适合“公网短平快首屏型”接口，至少在更多真机样本出来前应保持保守。

## 2) 证据边界

1. 主要证据来自以下文档：
   - `docs/dio_rust_test/network_benchmark_aggregation_2026-02-24.md`
   - `docs/dio_rust_test/network_large_json_validation_2026-02-24.md`
   - `docs/dio_rust_test/network_route_strategy_2026-02-24.md`
   - `docs/dio_rust_test/network_public_remote_sample_2026-03-13.md`
2. 这些结论大部分是 L1 传输层结论，只覆盖请求层时延、吞吐、错误、fallback、排队，不覆盖“Rust 解析对象后直接跨 FFI 回 Dart 模型”的完整对象回传成本。
3. 2026-03-13 的公网样例只是 `host_windows + ethernet + public_remote + smoke`，不是 Android / iOS 真机长期稳态结论。

## 3) 哪些业务接口适合优先切 Rust

### 3.1 大包体下载类接口

适合的业务形态：

1. 离线包、资源包、启动素材包、音视频分片、图片原图、富媒体详情数据。
2. 返回体明显大于普通配置接口，主要成本在网络传输和字节搬运。

原因：

1. `large_payload` 场景里 Rust 明显领先：
   - Dio：`p95=128ms`，`87.78 req/s`
   - Rust：`p95=45ms`，`272.73 req/s`
2. 这类场景的核心价值不是“Rust 帮你解析业务对象”，而是“Rust 通道把传输和 bytes-first 回传做得更快”。

建议：

1. 优先从只读 GET、可缓存、可重试的下载接口开始切。
2. 接入后重点看 `adapterCost`、吞吐、fallbackCount、真实首包耗时。

### 3.2 大 JSON 读取类接口

适合的业务形态：

1. 大型配置快照、字典表、首页聚合数据、推荐流首批大列表、搜索预置数据。
2. 服务端返回 JSON 很大，但客户端仍然是 Dart 侧 `jsonDecode` / model build。

原因：

1. `large_json` 两组结果里 Rust 都领先：
   - `json_decode`：Dio `159ms / 58.08 req/s`，Rust `80ms / 90.23 req/s`
   - `json_model`：Dio `149ms / 58.06 req/s`，Rust `81ms / 90.84 req/s`
2. 当前实现测到的是“Rust 请求 + bytes/bodyFilePath 回 Dart + Dart decode/model”，不是“Rust 解析对象再跨 FFI 回 Dart 模型”。
3. 也就是说，Rust 优势成立的前提是：你走的是 bytes-first 路径，而不是指望 Rust 在当前实现里替你完成复杂对象桥接。

建议：

1. 优先用于“大 JSON 拉取但仍由 Dart 解码”的读取接口。
2. 如果后续要引入“Rust 端解析对象后回 Dart 模型”，应单独补 benchmark，不应直接复用本文结论。

### 3.3 批量并发预拉取类接口

适合的业务形态：

1. 冷启动后并发预取多个只读接口。
2. 列表页进入前预拉取多个卡片源。
3. 弱网下有可能出现重试堆积的只读接口。

原因：

1. `jitter_latency` 的多轮结果说明，Rust 在并发 `c4/c8` 时优于 Dio。
2. 当 `maxInFlightTasks >= 32` 时，Rust 在 `c16/c32` 也能恢复领先。
3. 这说明 Rust 不是不能做高并发，而是要先把并发闸门调到合适范围，否则会被排队放大拖垮。

建议：

1. 这类接口可以切 Rust，但前提是明确把 `maxInFlightTasks` 维持在 `>=32` 并完成真机复验。
2. 若当前线上配置仍低于 `32`，就只让低并发读请求优先走 Rust，不要贸然把高并发预拉取全量切过去。

### 3.4 可回退的幂等读接口

适合的业务形态：

1. 首页配置、Feed 拉流、详情读取、推荐请求、字典拉取。
2. 方法幂等，失败后允许安全回退到 Dio。

原因：

1. 当前网关设计本来就把 Dio 作为自动 fallback 和总开关回切通道。
2. 对于幂等读请求，Rust 更适合作为“主通道 + Dio 兜底”的灰度对象。

建议：

1. 先切幂等 GET / HEAD。
2. 保留 `enableFallback=true` 和 `enableRustChannel` 总开关。
3. 观测稳定后再考虑扩大覆盖面。

## 4) 哪些业务接口暂不建议默认切 Rust

### 4.1 公网首屏关键小请求

典型接口：

1. 启动首屏配置。
2. 登录态校验。
3. 首屏最小骨架数据。
4. 搜索建议、轻量推荐、小型统计接口。

原因：

1. loopback 的 `small_json` 里 Rust 很强，但 2026-03-13 公网 smoke 样例没有稳定复现这个优势。
2. 同一天公网短请求样本里，Rust 首跑和 rerun 波动明显，说明冷连接、连接复用或公网路径适配行为还不够稳定。
3. 这类接口通常对首包时延更敏感，单次抖动就会直接体现在用户体验上。

建议：

1. 在 Android / iOS 真机、Wi-Fi / 4G / 弱网样本补齐前，先保留 Dio。
2. 如果一定要灰度 Rust，应避开“首屏唯一依赖”，先做旁路观测或小流量灰度。

### 4.2 高抖动高并发且并发闸门未放开的接口

典型接口：

1. 同时拉很多卡片源的首页。
2. 冷启动批量预取。
3. 弱网反复重试后形成的并发堆积。

原因：

1. `jitter_latency` 结果已经证明，当 `maxInFlightTasks=12` 时，Rust 在 `c16/c32` 会被排队放大拖慢。
2. 这类接口一旦踩到容量阈值，问题不是平均值不好看，而是 p95/p99 直接劣化。

建议：

1. 若线上仍是低 `maxInFlightTasks`，此类接口优先走 Dio。
2. 只有在 `>=32` 并且真机复验通过后，才建议切回 Rust。

### 4.3 非幂等写接口

典型接口：

1. 下单、支付、提交表单、发帖、状态变更、扣费、发奖。

原因：

1. 这些接口不是不能走 Rust，而是不应该把“自动 fallback”当作默认保护手段。
2. 当前回退规则要求请求幂等，或显式提供 `Idempotency-Key`。

建议：

1. 写接口默认继续保守接入。
2. 若业务必须接 Rust，请先单独补充幂等策略与错误恢复设计，再做专项验证。

## 5) 业务路由建议

### 5.1 第一批优先切 Rust

1. 大包体 GET。
2. 大 JSON GET，且仍为 Dart 侧 decode/model。
3. 批量只读预拉取。
4. 可缓存、可重试、可自动 fallback 的幂等读接口。

### 5.2 第二批条件式切 Rust

1. 中等体积 JSON。
2. 有一定并发，但依赖 `maxInFlightTasks>=32` 才能稳定占优的接口。
3. 非首屏唯一依赖、允许灰度观测的读取接口。

### 5.3 暂时保留 Dio

1. 公网首屏关键小请求。
2. 对冷启动单次抖动极敏感的小接口。
3. 非幂等写请求。
4. 尚未在真机网络剖面下补齐证据的高风险接口。

## 6) 一个简单的业务判断法

如果某个接口同时满足以下条件，优先考虑 Rust：

1. `GET/HEAD` 幂等读。
2. 返回体大，或者存在并发预拉取。
3. 客户端主要是拿 bytes，再在 Dart 里消费。
4. 允许保留 Dio fallback 和总开关回切。
5. 已在目标设备和目标网络剖面上做过样本验证。

如果某个接口更接近以下特征，优先保留 Dio：

1. 小请求、小响应、首屏强依赖。
2. 公网下对首包时延极敏感。
3. 写操作或幂等性不清晰。
4. 当前还没有真机复验或稳定性证据。

## 7) 当前建议的落地顺序

1. 先切：资源下载、大 JSON、批量只读预拉取。
2. 再切：中等体积读接口，但要求完成真机网络剖面验证。
3. 最后评估：公网首屏短请求、小接口、写接口。

## 8) 后续补证据建议

1. 用同一套固定入口补 `android_real` / `ios_real`。
2. 至少覆盖 `wifi` / `4g` / `weaknet`。
3. 对短请求增加“首跑 + 立即 rerun”成对采样，判断是否存在明显冷连接波动。
4. 对目标业务接口补充 30~120 分钟长稳压测，而不是只看 smoke。
